# Informe de Estado y Documentación de Funcionamiento: mmex_lib
**Fecha:** 25 de febrero de 2026
**Estado:** Arquitectura Base Estable (v0.1.0-alpha)

## 1. Resumen de la Arquitectura Implementada

Se ha establecido una arquitectura de **Domain-Driven Design (DDD)** 100% sincrónica y sin dependencias de runtimes externos, garantizando portabilidad máxima (FFI, CLI, Desktop).

### Capas de la Aplicación:
*   **Domain (Núcleo):** 
    *   `src/domain/types.rs`: Definición de Newtypes (`AccountId`, `Money`) para evitar Primitive Obsession.
    *   `src/domain/models.rs`: Entidades ricas (`Account`) con validación de tipos.
    *   `src/domain/repositories.rs`: Interfaces (traits) de persistencia que desacoplan el dominio de la infraestructura.
*   **Infrastructure (Persistencia):**
    *   `src/infrastructure/db_executor.rs`: Implementación del patrón *Executor* para manejar Conexiones y Transacciones de forma polimórfica.
    *   `src/infrastructure/repositories.rs`: Implementación de repositorios SQL usando `sea-query` para mapeo dinámico.
    *   `src/infrastructure/mapper.rs`: Lógica de transformación del esquema legacy (Mayúsculas/Strings) a modelos Rust.
*   **Services (Aplicación):**
    *   `src/services/account_service.rs`: Orquestador de lógica de negocio para cuentas.
*   **API (Fachada):**
    *   `src/api/context.rs`: El `MmexContext` como punto de entrada único (Facade) gestionando la conexión SQLite + SQLCipher.

## 2. Hitos Técnicos Alcanzados
1.  **Gestión de Precisión:** Uso mandatorio de `rust_decimal` para todos los cálculos financieros.
2.  **Escalabilidad de Identificadores:** Migración total a `i64` para todos los tipos de ID para soportar bases de datos con grandes volúmenes de datos.
3.  **Referencia de Esquema:** Adopción de `tables.sql` como fuente de verdad única para el mapeo de infraestructura.
4.  **Desacoplamiento de Transacciones:** Los servicios pueden ejecutar lógica atómica sin conocer detalles de `rusqlite` gracias al trait `DbExecutor`.
3.  **Alineación de Dependencias:** Resolución de conflictos entre `sea-query` (v0.32) y `sea-query-rusqlite` (v0.7).
4.  **Validación Empírica:** Implementación de tests de integración con bases de datos en memoria (`:memory:`) que verifican el ciclo completo de datos.

## 3. Guía de Funcionamiento
El flujo estándar de la librería sigue este patrón:
1.  `MmexContext::open(...)` establece la sesión cifrada.
2.  El contexto expone servicios específicos (ej. `context.accounts()`).
3.  Los servicios consultan al repositorio, el cual construye SQL seguro con `sea-query`.
4.  El mapeador transforma los resultados legacy en entidades de dominio limpias que se devuelven al cliente.
